# 路线图

WutherCore 采用“能力完成后再稳定接口”的节奏，不用日期代替质量承诺。路线图按工程结果排序，优先级会根据安全问题、平台回归和真实使用反馈调整。

## 当前重点

### 协议与互操作

- 为主要出站协议补充可重复的客户端/服务端兼容测试；
- 明确每个协议的 TCP、UDP、复用和传输组合；
- 对第三方协议升级建立回归样例；
- 把不完整能力从“静默降级”改为明确错误或状态说明。

### 透明代理可靠性

- 完善 Windows、Linux、macOS 和 Android 的启动/停止验证；
- 覆盖路由、防火墙和接口创建的部分失败回滚；
- 降低与其他 VPN、容器网络和多网卡环境的冲突；
- 为权限不足和系统依赖缺失提供可操作诊断。

### 配置与 API 稳定性

- 收敛 `version: 1` 的字段、默认值和迁移规则；
- 为公开 API 建立稳定响应模型和兼容测试；
- 完成常用 Mihomo 配置的迁移覆盖；
- 为错误信息补充字段路径和修复提示。

### 安全与供应链

- 完成 [CodeQL 初始告警](https://github.com/MiChongs/WutherCore/issues/9) 分类；
- 持续审查依赖变更和高危漏洞；
- 检查凭据、订阅和网络元数据的日志边界；
- 为管理 API、配置解析和规则集输入补充对抗性测试。

### 文档与发布

- 让功能矩阵与代码实现同步；
- 提供可复现的跨平台构建产物和校验信息；
- 为配置升级和破坏性变化维护迁移说明；
- 在 Release Notes 中明确已知限制和验证平台。

## 1.0 完成标准

1. 核心配置与管理 API 有明确兼容策略。
2. 主要协议和平台路径有持续集成或可重复的手动验证记录。
3. 透明代理的安装、失败与退出均有可靠回滚。
4. 安全扫描告警有可解释基线，没有未跟踪的高风险问题。
5. 发布产物可追溯、可校验，并带升级与回滚说明。
6. README、功能矩阵、配置、API 与排错文档和实现一致。

## 不在内核范围

- 桌面或移动端 GUI；
- 机场、订阅或节点服务；
- 绕过服务端授权或协议访问控制；
- 对所有第三方 Dashboard 私有接口的完全兼容。
